How each transaction is classified during import
1

User-defined Rules (highest priority)

Rules are checked in priority order (1 → n). First rule whose ALL conditions match wins immediately. No further checks done.

↓ no rule matched
2

Income Source matching

Description checked against user's saved Income Sources (Rahul Sharma → Rent, etc.). Overrides built-in rules when matched.

↓ no source matched
3

Built-in keyword rules

50+ merchant patterns: ZOMATO→Food, AMAZON→Shopping, UPI/DR→Expense, SALARY→Income, SWEEP IN→Investment, etc.

↓ no keyword matched
4

Heuristics + large deposit check

Debit column → Expense. Credit column → Income. Large unclassified credit (≥₹10,000) gets flagged with ⚠️ for user review.

↓ still ambiguous
5

Flag for manual review

Row shown with ⚠️ in preview table. User must pick type + category before confirming import.

🔁 Retroactive application

When creating a rule, you can choose "Apply to all existing transactions too." The system will re-classify matching expenses already in the DB — updating their category and type automatically. A summary is shown: "12 existing transactions updated."

💡 One-click rule creation from preview table

After manually fixing a row in the preview table, a "Save as rule" button appears. Clicking it pre-fills a new rule with the description keyword and the category you chose — so you never have to do it again.
